# The Javascript Interview Questions They Should Ask

---

The Javascript Interview Questions They Should Ask

Ok — so I’ve done quite a few interviews over the years, probably at least 20 in all. I’ve also interviewed about a dozen candidates during this time as well and I hate to admit, I’ve fallen in the trap of asking the same basic flavor of Javascript questions I’ve been asked. I think Kyle Simpson probably dies a little bit inside each time I’ve asked a candidate to find the duplicate in an array 🤦‍♂.

Some questions have been posed so often they become trite: finding a duplicate, FizzBuzz, 3sum, 2sum etc. These toy problems can offer some insight into how a candidate approaches a problem but more often than not, they really just show how much time that person has dedicated to studying Leetcode. To be fair, there are some times you’ll need to traverse a tree-like structure or think of an efficient way to store data with constant time look up. I also think that studying data structures and common algorithms is an excellent trait and shows dedication and willingness to learn which are qualities most engineering teams want their members to possess. But let’s get real for a minute here — your day to day work as a JS dev is going to involve a lot less Djikstra and a whole lot more npm install

What are these tests really looking to ascertain? That the candidate will know enough JS to successfully engage with your codebase? What level is their mastery of Javascript as a whole (do they understand closure, inheritance, functional programming)? or is it to see how well they can communicate their thought process under the awkward circumstance of being in a room with strangers? Maybe some combination of all the above?

Whether it’s Google, Facebook or your company, which probably thinks it is the next Google or Facebook — I’ve seen questions posed during interviews that boil down to being clever with JS while understanding common CS fundamentals or worse, obscure JS concepts that you could easily look up (what happens if you run isNaN === isNaN ). There’s got to be a better way.

To get a general sense of how well a person understand’s JS and also how they will approach building something, I think the following questions are a much better indicator of how well someone will do on your typical front-end focused team:

Using only HTML, CSS and JS, display this array of data on the screen, paginated so only 10 items are visible at a time.

Let’s make a chart using some library you’ve never used (HighCharts, Victory or D3 if you want to be a jerk) — add some click events if time allows.

For React-centered roles, refactor this class based component using lifecycle methods like componentDidMount or componentDidUnmount to a functional component using hooks

Merge these two arrays of objects by matching them another object with the same id:

5. Here’s a third party API with a key — create a UI to consume the data from this UI and make it look like this (*shows picture of simple UI to candidate)

Obviously these suggestions won’t work for all JS developer roles. Perhaps you really do need someone with strong CS fundamentals and expert level understanding of Typescript or some obscure JS framework like Vue 😉 because they will be your first engineering hire or building the initial product using this technology.
