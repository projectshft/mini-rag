# Sparse and Dense Vectors in RAG

## Introduction

When building RAG systems, you have two main approaches to representing text for search: **dense vectors** and **sparse vectors**. Many production systems use both in a hybrid approach.

## Dense Vectors

**What they are:**
- Fixed-size embeddings (e.g., 512, 1536 dimensions) generated by neural networks
- Every dimension has a value (hence "dense")
- Capture semantic meaning and context

**Example:** The embedding for "king" and "monarch" will be similar even though they share no words.

**Strengths:**
- Excellent for semantic similarity
- Handles synonyms, paraphrasing, and conceptual matches
- Works across languages

**Weaknesses:**
- Can miss exact keyword matches
- "Black swan" might not match "black swan" if the model thinks it's about rare events vs. actual birds
- Less interpretable (you can't see "why" something matched)

## Sparse Vectors

**What they are:**
- High-dimensional vectors (often 30k+ dimensions) with mostly zeros
- Based on keyword matching (like BM25 or TF-IDF)
- Each dimension represents a specific term

**Example:** "React hooks" creates a vector with non-zero values only for positions corresponding to "react" and "hooks".

**Strengths:**
- Excellent for exact keyword matches
- Preserves important terminology (product names, technical terms, IDs)
- Interpretable - you know exactly which terms matched

**Weaknesses:**
- Misses semantic similarity
- "automobile" won't match "car"
- Struggles with typos and variations

## Why Use Hybrid Search?

Combining sparse and dense vectors gives you the best of both worlds:

1. **Precision + Recall**: Dense vectors find conceptually similar content, sparse vectors ensure exact matches aren't missed

2. **Domain-Specific Terms**: Technical documentation often has specific terminology where exact matches matter
   - Dense: Finds documents about "state management" when you search for "managing application state"
   - Sparse: Ensures "useState" matches exactly "useState" and not just "state"

3. **Better User Experience**: Users sometimes search with exact phrases they remember, other times with natural questions

## Example Use Case

Search query: "How to prevent re-renders in React?"

- **Dense vectors** find: Documents about React performance, memoization, optimization
- **Sparse vectors** find: Documents mentioning "re-render", "React.memo", "useMemo"
- **Hybrid** combines both for comprehensive results

## Pinecone Implementation

Pinecone supports hybrid search natively, making it straightforward to implement:

- **Documentation**: [Pinecone Hybrid Search Guide](https://docs.pinecone.io/guides/data/understanding-hybrid-search)
- **How it works**: You provide both dense embeddings and sparse vectors in the same upsert operation
- **Querying**: Combine both vector types with an `alpha` parameter to control the balance

**Basic concept:**
```javascript
// Upsert with both dense and sparse vectors
await index.upsert([{
  id: 'doc1',
  values: [0.1, 0.2, ...], // Dense embedding
  sparseValues: {
    indices: [5, 10, 15],
    values: [0.5, 0.3, 0.2]
  },
  metadata: { text: '...' }
}]);

// Query with hybrid search
const results = await index.query({
  vector: denseEmbedding,
  sparseVector: sparseVector,
  topK: 10,
  alpha: 0.5 // 0 = pure sparse, 1 = pure dense, 0.5 = balanced
});
```

## When to Implement Hybrid Search

**Consider hybrid search when:**
- Your domain has specific terminology that must match exactly
- Users search both with natural questions and exact terms
- You need high precision for technical documentation
- Missing exact matches causes poor user experience

**Stick with dense-only when:**
- You're just getting started (keep it simple first)
- Your content is conversational without critical exact-match terms
- The added complexity isn't justified by improved results
