Tool-Calling vs Workflows & Building a Selector Agent

The Problem

You have multiple knowledge bases. How does your RAG system know which one(s) to search?

Your Knowledge Bases:

LinkedInPosts MediumArticles ScientificPapers
  • "What are good LinkedIn post ideas?" → Search LinkedInPosts
  • "Latest transformer research" → Search ScientificPapers
  • "Content writing tips" → Search LinkedInPosts + MediumArticles

Two Approaches to Solve This

🔧 Tool-Calling

LLM decides on the fly which tools/functions to invoke

  • Dynamic decision-making
  • LLM controls the flow
  • Can be unpredictable
  • Good for open-ended tasks

📋 Workflow

Pre-defined steps executed in a deterministic order

  • Predictable execution
  • Code controls the flow
  • Easier to debug
  • Good for structured tasks

🔧 Tool-Calling: How It Works

You give the LLM a toolbox. It decides what to use.

User Query → 🧠 LLM
picks from toolbox ↓
🔍 searchDB
📧 sendEmail
📝 createDoc
🌐 fetchURL
💾 saveFile
📊 runQuery
↓ executes tools, maybe loops back
Response

The LLM is in control — it decides what, when, and how many times

✅ When Tool-Calling Works Well

Good For

  • Open-ended exploration
    User: "Research this topic and summarize"
  • Personal assistants
    Calendar, email, tasks — flexible intent
  • Developer tools
    Claude Code, Cursor — user supervises
  • Low-stakes tasks
    Mistakes are cheap to fix

Bad For

  • Production pipelines
    Need predictable, testable behavior
  • High-stakes decisions
    Financial, medical, legal
  • Cost-sensitive apps
    LLM might loop and burn tokens
  • Latency-sensitive apps
    Multiple round-trips = slow

🏢 Why Most Companies Need Workflows

🔒

Predictability

Same input → same steps
Every. Single. Time.

🐛

Debuggability

Step failed? You know exactly where and why

💰

Cost Control

Fixed # of LLM calls
No runaway loops

⚡

Speed

Parallel steps, no back-and-forth

🧪

Testability

Unit test each step
Mock dependencies

📋

Compliance

Audit logs, approvals
Regulatory needs

Rule of thumb: If you're building for users (not developers), use workflows.

💬 From Anthropic

"When building applications with LLMs, we recommend finding the simplest solution possible, and only increasing complexity when needed. This might mean not building agentic systems at all."

"When more complexity is warranted, workflows offer predictability and consistency for well-defined tasks, whereas agents are better when flexibility and model-driven decision-making are needed at scale."

"For many applications, however, optimizing single LLM calls with retrieval and in-context examples is usually enough."

— Anthropic, "Building Effective Agents"

🎓 Fine-Tuning vs RAG

Two ways to make LLMs smarter about your domain

🔧 Fine-Tuning

Train the model on your data

  • Changes model behavior/style
  • Learns patterns, tone, format
  • Knowledge baked into weights
  • Can't easily update

Use when: You need a specific voice, style, or behavior the base model can't do

📚 RAG

Give the model your data at runtime

  • Provides knowledge/facts
  • Retrieves relevant context
  • Knowledge in vector DB
  • Easy to update anytime

Use when: You need current facts, docs, or domain knowledge

Today: We're building RAG (the more common pattern)

📝 Fine-Tuning: Quick Look

(We won't implement this today — just understanding the concept)

Training data teaches the model a specific voice/style:

// linkedin_training.jsonl — example format { "messages": [ { "role": "system", "content": "You are a brutally honest career coach..." }, { "role": "user", "content": "Write a post about tutorial hell." }, { "role": "assistant", "content": "Turn off the tutorial. Open the code editor..." } ] }

Fine-tuning is for: Changing HOW the model responds (tone, format, style)

RAG is for: Giving the model WHAT to know (facts, docs, context)

📋 Workflow Pattern

Structured steps with LLM making specific decisions at each stage

// Step 1: Selector Agent picks indexes const { indexes, query } = await selectorAgent(userMessage); // Step 2: Search selected indexes (deterministic) const results = await searchIndexes(indexes, query); // Step 3: Generate response with context const response = await generateResponse(results, messages);

Benefit: Predictable flow, LLM focused on narrow decision

🎯 What We're Building

A Selector Agent using the Workflow Pattern

User Query → Selector Agent → 0-3 Indexes → RAG Response

The Selector Agent outputs structured data (not free text)

// Selector Agent Output Schema { indexes: ["LinkedInPosts", "MediumArticles"], // 1-3 indexes query: "optimized search query for retrieval" }

🔒 Structured Output with Zod

Force the LLM to return valid, typed data

const indexSchema = z.enum([ 'LinkedInPosts', 'MediumArticles', 'ScientificPapers' ]); const indexSelectionSchema = z.object({ indexes: z.array(indexSchema).min(1).max(3), query: z.string() }); // OpenAI will ONLY return data matching this schema const response = await openai.chat({ response_format: zodTextFormat(indexSelectionSchema) });

📚 Few-Shot Examples

Guide the LLM with examples of correct behavior

// Examples in our system prompt: User: "What are good LinkedIn post ideas for developers?" Output: {indexes: ["LinkedInPosts"], query: "developer post ideas engagement"} User: "Latest research on transformer architectures" Output: {indexes: ["ScientificPapers"], query: "transformer architecture research"} User: "How do I write engaging content?" Output: {indexes: ["LinkedInPosts", "MediumArticles"], query: "engaging content tips"}

🎯 Key Takeaways

  • Tool-Calling: LLM decides flow dynamically — flexible but unpredictable
  • Workflows: Code controls flow, LLM makes narrow decisions — predictable
  • Selector Agent: Uses structured output to pick 1-3 indexes
  • Zod Schemas: Guarantee valid, typed LLM responses
  • Few-Shot: Examples guide the LLM's decision-making

Why this matters: Multi-index RAG lets you build specialized knowledge bases while keeping a unified chat interface.
